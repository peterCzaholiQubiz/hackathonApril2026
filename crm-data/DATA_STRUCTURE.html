Core CRM Entities
The master records that anchor all other data. Organizations are parties (customers, brokers, collectives). Contracts bind organizations to energy products. Connections are the physical delivery points (EAN codes).
🏢
Organizations
ERPSQLServer/Organizations.csv
Core
Master table of every party in the system — customers, collectives (multi-site groups), deal counterparties, brokers, and head organisations. The primary anchor for all contract, contact, and invoice data. Names are pseudonymised hash tokens.
FieldTypeNotes

OrganizationTypeIdFKint→ OrganizationTypes. 2=Customer, 5=Collective, 6=Company, 7=Broker
DebtorReferencestringExternal accounting key; nullable
NamestringPseudonymised hash
UserNameModifiedstringAudit: last modifier
TransStartDatedatetimeRecord creation / onboarding timestamp
Related to
→ OrganizationTypes → Contracts (via join) → OrganizationContacts → InvoiceArchive

Contracts
ERPSQLServer/Contracts.csv
Core
Contract master data — one row per contract record. Includes both long-running customer contracts (ContractType="Customer") and their sub-period derivatives (ContractType="Period"). EndDate of 9999-12-31 means open-ended. CurrentAgreedAmount is the monetary value of the contract.

EndDatedate9999-12-31 = open-ended
CurrentAgreedAmountdecimalContract monetary value (EUR)
UserCommentstringFree-text annotation
Related to
→ ProductTypes → ContractPrice → PriceProposition → DQE-Prijzen → ReportingJoin

Connections
ERPSQLServer/Connections.csv
Core
Master table of every physical energy connection point — one row per metering point. The EAN (European Article Number) is the 18-digit code that uniquely identifies each electricity or gas delivery point in the Dutch/Belgian grid. DeliveryType distinguishes supply (LDN) from return-feed (ODN — e.g. solar panels).
FieldTypeNotes
ConnectionIdPKintInternal connection identifier
EANKEYstringPseudonymised 18-digit grid code. Universal join key to all timeslice/meter files
ProductTypestring"Electricity" | "Gas"
DeliveryTypestring"LDN" (supply) | "ODN" (return/feed-in) | "NA"
ConnectionTypeIdFKint→ ConnectionTypes (CHP, WindTurbine, MainConnection…)
DescriptionstringPseudonymised
ClientReferencestringCustomer's own reference code
Related to
→ ConnectionTypes → MeterRead 1–8 → ConnectionMeterReads → All Timeslices → DQE-Captars → DQE-Prijzen → ERPMRE → ASU001 → CPY001 → ReportingJoin
Contact & Interaction History
Records of every interaction between the supplier and their customers or connections. Subject field carries semantic value (e.g. "Cancellation" is a churn signal). Free-text report fields are pseudonymised hashes.

Pricing
Three complementary files that together reconstruct the full price book applied to each contract: agreed contract tariffs, catalogue proposition tariffs, and the full historical price-component ledger per connection.

Meter & Consumption Data
Four related datasets capturing energy consumption at different granularities. Meter Read 1–8 and ConnectionMeterReads are the primary consumption sources. ERPMRE is the lowest-level event fact table. ASU001 holds the official annual standard usage figures.
🔢
Meter Read × 8 files
ArchievingSolution/Generic/Meter Read_1.csv … _8.csv
Meter
Full meter-reading history per EAN across 8 split files (split for volume). Each row records a metered quantity for a period. Dual-tariff electricity meters produce two rows per period (UsageLow=off-peak, UsageHigh=peak). Direction distinguishes grid supply from solar feed-in. All 8 files have identical schema and must be scanned together.
FieldTypeNotes
EANUniqueIdentifierFKstring→ Connections.EAN
StartDate / EndDatedatePeriod covered by this reading
ConsumptiondecimalDelta consumption over the period
Position / PreviousPositiondecimalRaw meter counter values at end/start of period
Sourcestring"DVEP" | "Customer" | grid operator
Qualitystring"Estimated" | "Customer" | "Actual"
UsageTypestring"UsageLow" (off-peak) | "UsageHigh" (peak)
Directionstring"Consumption" | "Production" (solar feed-in)
MeterIdentifierstringPhysical meter serial number
MeterFactordecimalCT multiplier for scaled meters

The lowest-level meter fact table — one row per metered quantity per individual read event. Includes kWmax (peak demand) in addition to standard usage types. Source of record for both normal consumption and peak-demand values. Amount can be negative for production (solar). This is the largest table in the dataset.

ASU001 — Annual Standard Usage
ERPSQLServer/[ValueAQuery] ASU001.csv
Meter
Official SJV (Standaard Jaar Verbruik) values per EAN — the annual standard usage figures required by the Dutch market for balance reconciliation. Split into four components: peak/off-peak on both consumption and production sides. Used by balance-responsible parties to estimate hourly consumption for unmetered connections.

EAEnergyProductionNettedPeakdecimalAnnual peak feed-in

Timeslices & Network Tariff History
All time-sliced history tables share the same 4-column schema: EANUniqueIdentifier, StartDate, EndDate, Description. They record how a connection's classification changed over time. DQE-Captars is richer — it stores the full network tariff breakdown for each classification period.
⏱️
Timeslices × 7 files
ArchievingSolution/Generic/Timeslices - *.csv
Timeslice
Seven files, all with identical schema, each recording a different classification dimension for each EAN over time. All use StartDate/EndDate validity windows. Key files for risk scoring:
FileDescription field contains
CaptarCodeGrid-operator tariff category (affects regulated charges)
ConnectionType"Levering" (supply) | "Teruglevering" (solar return)
EnergyDeliveryStatus"Aangesloten" (live) | "Afgesloten" (disconnected)
PhysicalStatus"In bedrijf" (operational) | "In aanleg" (under construction)
ProfileLoad profile code (1A, 1B, 2A, 3A, 4A…) for unmetered allocation
ResidentialFunctionDwelling classification — drives VAT and regulated tariff applicability
UsageType"Kleinverbruik" (small, regulated) | "Grootverbruik" (large, negotiated)
Shared schema fieldNotes
EANUniqueIdentifierFK→ Connections.EAN
StartDate / EndDateValidity window; 9999-12-31 = current
DescriptionDutch label for the classification value

Reporting & Cross-Entity Joins
Flattened join tables pre-computed for reporting. These denormalize multi-table joins into single rows, answering "which customer was behind this EAN under this contract, through which broker, billed to which debtor" without requiring complex joins at query time.
🔗
Contract–Customer–Connection–BrokerDebtor
ERPSQLServer/Contract-Customer-Connection-BrokerDebtor.csv
Reporting
The primary reporting join. One row per contract-per-connection, answering: which customer held this EAN under which contract, via which broker, billed to which debtor, between these dates. Names are pseudonymised. Essential for linking Organizations (by OrganizationId) to Contracts and Connections, since no direct FK exists between them.

DebtorNumber / DebtorNamestringLegal billing party; pseudonymised
Invoice Archive Index
Two-part index mapping each archived invoice to the folder on the archive drive where its supporting documents live. Split purely because of row volume — both files have identical columns and must be scanned together for any invoice lookup.
🗃️
Look-up Customer Data × 2 files
ArchievingSolution/Look-up Customer Data_1.csv + _2.csv
Archive
The primary entry point when locating physical invoice documents for a specific invoice, debtor, or customer. The "Collective name" is a hashed group key used to keep related parties together. Both files must be searched — they are a continuous index split at a row boundary. No invoice amounts or dates are stored here; this is purely a location index.
FieldTypeNotes
Collective namestringHashed group key; links related parties. Joins across files work.
Customer numberFKstring→ Organizations (type=Customer) via CustomerNumber
Debtor numberstringLegal billing party (one customer can have multiple debtors over time)
Invoice numberKEYstringInvoice identifier — primary lookup key
File locationstringFull path on archive drive, e.g. T:\Archive data\Debtor\29355\1908981
Note
Scan both files for complete results No amount / date columns
Lookup / Reference Tables
Small enumeration tables used as foreign key targets. Stable — rarely changes. All follow the same pattern: an ID, an English code (Description), and a Dutch UI label (PresentationDescription).
